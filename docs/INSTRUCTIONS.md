# Canonical Agent Instructions

Lex can keep one repository-owned instruction source and project it into host-specific files such
as `.github/copilot-instructions.md` and `.cursorrules`.

This feature is optional. It does not run during the basic Frame pilot, and you do not need the
broader `lex init` workspace bootstrap to adopt it.

## The workflow

```text
.smartergpt/instructions/lex.md     ← humans review this canonical source
                 │
                 │ lex instructions generate
                 ▼
.github/copilot-instructions.md    ← Lex-managed block
.cursorrules                       ← Lex-managed block
```

Lex owns only the content inside its markers. Content outside those markers remains host-specific
and is preserved.

## 1. Initialize the instruction surface

```bash
lex instructions init
```

This scaffolds the canonical source, `lex.yaml`, and selected host targets. Choose hosts explicitly
when useful:

```bash
lex instructions init --targets copilot,cursor
lex instructions init --project-root /absolute/path/to/repository
```

`--force` overwrites existing scaffolded files. Review its effects before using it in a repository
that already has agent instructions.

## 2. Edit the canonical source

The default source is `.smartergpt/instructions/lex.md`. Keep it repository-specific and concise:

- explain important ownership and architecture boundaries;
- name the validation commands an agent should run;
- describe intentional working-tree or release conventions;
- link to detailed references instead of copying them;
- avoid model-specific phrasing unless the rule truly belongs to one host;
- never include credentials, user-specific absolute paths, or ephemeral timestamps.

A small example:

````markdown
# Repository Instructions

## Boundaries

- Keep authentication validation in `api/middleware`.
- Access credential storage only through `services/auth`.

## Validation

```bash
npm test
npm run type-check
```

## Continuity

Create a Lex Frame at a meaningful handoff:

```bash
lex remember \
  --summary "What changed and why" \
  --next "The next concrete action" \
  --modules unscoped
```
````

## 3. Preview and generate projections

Preview without writing:

```bash
lex instructions generate --dry-run
```

Write the selected projections:

```bash
lex instructions generate
```

Operate on a different repository or configuration explicitly:

```bash
lex instructions generate \
  --project-root /absolute/path/to/repository \
  --config /absolute/path/to/lex.yaml
```

Generation exits `0` on success and non-zero on failure. The current command uses `0`/`1`; do not
build automation around a separate validation exit code.

## 4. Check drift in CI

`check` is the non-mutating CI surface:

```bash
lex instructions check
```

It exits non-zero when the projections do not match the canonical source. A minimal GitHub Actions
step is:

```yaml
- uses: actions/setup-node@v4
  with:
    node-version: "24"
- run: npm install --global @smartergpt/lex
- run: lex instructions check
```

Do not run `generate` followed by a non-failing `git diff` in CI; that mutates the checkout and can
silently accept drift.

## Configuration

The default `lex.yaml` shape is:

```yaml
version: 1

instructions:
  canonical: .smartergpt/instructions/lex.md
  projections:
    copilot: true
    cursor: true
```

| Option | Purpose |
|---|---|
| `instructions.canonical` | Repository-relative canonical instruction source |
| `instructions.projections.copilot` | Project into `.github/copilot-instructions.md` |
| `instructions.projections.cursor` | Project into `.cursorrules` |

An explicit configuration is preferable when the desired host set should remain stable across
developer machines.

## Marker contract

Generated blocks use one ordered marker pair:

```markdown
<!-- LEX:BEGIN -->
<!-- This block is auto-generated by `lex instructions generate`. Do not edit manually. -->

[canonical instruction content]

<!-- LEX:END -->
```

- Edit the canonical source, not the generated block.
- Keep at most one marker pair in each target.
- Preserve `LEX:BEGIN` before `LEX:END`.
- Lex preserves content outside the pair.

If generation appends instead of replacing content, inspect the target for missing, duplicated, or
misordered markers before running it again.

## Troubleshooting

### No target was generated

Confirm that the canonical source exists, the desired projection is enabled in `lex.yaml`, and the
target host was initialized. Re-run `lex instructions init --targets ...` only after reviewing
whether existing files may be changed.

### A projection is stale

```bash
lex instructions check
lex instructions generate --dry-run
```

Fix canonical/configuration errors, review the preview, then run `generate` explicitly.

### A different repository is being modified

Use `--project-root` with an absolute native path. In a trusted multi-workspace host, inject the
authorized project root rather than deriving it from ambient `cwd` or environment state.
